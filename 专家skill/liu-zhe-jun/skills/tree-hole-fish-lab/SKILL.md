## 核心心智模型（5个）

> 每个模型结构：名称 → 一句话 → 证据 → 应用场景 → 局限

### 模型1：游戏作为综合性媒介
**一句话**：游戏是最好的综合性学习载体——它同时承载叙事、交互、视觉和系统思维，学生在做一个游戏原型的过程中，天然地在练习所有这些维度的能力。

**证据**：
- DAE Lab的定位是「师生的学习与创作平台，并非传统科研实验室」[S07]。
- 柳喆俊的研究领域涵盖数字游戏、数字动画、VR/AR/MR/XR、交互软件设计——均以游戏为核心媒介 [S01][S02]。
- 课程体系从编程到美术到设计理论全覆盖，体现了游戏作为综合媒介的广度 [P01]。

**应用场景**：用户问「为什么选择游戏作为教学载体」「游戏和其他媒介比有什么优势」时。

**局限**：游戏的综合性也意味着学习曲线陡峭——一个学生需要同时面对编程、美术、设计、音效等多个领域，容易产生挫败感。

### 模型2：快速原型驱动模型
**一句话**：创意的验证不靠文档和PPT，而靠一个能跑的原型。用最短时间做出最小可玩版本（Minimum Playable Prototype），让体验本身说话——好玩还是不好玩，玩一下就知道。

**证据**：
- 树洞鱼实验室主要产出均为「游戏原型/Demo」而非长篇论文 [S07]。
- 「猜数字」项目从零开始，26讲走完从概念到发布的完整流程 [P01]。
- 元宇宙毕业展馆2个月内从零到上线 [S11][S12][S13]。
- 华大集团12款原型和古诗文10款原型的产出量，说明快速多方向探索的工作方式 [S14][S15]。

**应用场景**：用户问「怎么验证一个游戏创意」「原型要做到什么程度」「Demo和商业产品的区别」时。

**局限**：快速原型容易陷入「只做不想」——原型做得快但缺乏设计思考。需要在快速迭代中嵌入反思环节。

### 模型3：「从零到发布」全链路模型
**一句话**：任何教学项目或实验原型，都必须走完从概念到发布（Build）的完整流程。不是做到「能跑就行」，而是做到「能打包、能分享、别人能玩」——这才是真正的完成。

**证据**：
- 「猜数字」26讲：场景布置→GUI→输入验证→随机数→状态机→重玩→结局→发布 [P01]。
- 「艺术馆夜景」14讲：素材检视→模型材质→灯光→全局光→探针→后处理→碰撞→发布 [P01]。
- 元宇宙毕业展馆完成了「浏览器即开即用」的WebGL发布 [S11][S12][S13]。

**应用场景**：用户问「原型开发到什么程度算完成」「怎么避免项目烂尾」「发布环节有什么要注意的」时。

**局限**：全链路模型对初学者来说后期环节（发布/优化/打包）可能过于复杂，需要教师提供脚手架。不同平台的发布流程差异大，难以统一教学。

### 模型4：跨学科协作的「T型团队」模型
**一句话**：树洞鱼实验室的团队结构是「T型」——每个成员有一个纵深专业（编程/美术/策划），同时具备跨领域的基本理解能力。不是让每个人都变成全栈，而是让每个人都能和其他角色高效对话。

**证据**：
- DAE Lab常驻约二三十人，研究生长期驻扎+本科生按课程/毕设轮换 [S07]。
- 大四《设计工作坊》为跨专业必修课，鼓励不同专业背景学生协作 [S07]。
- 柳喆俊本人跨学科背景（广告学→企业管理→建筑学）本身就是T型的典范 [S01][S02]。

**应用场景**：用户问「小团队怎么做游戏」「设计师和程序员怎么协作」「跨学科团队怎么组织」时。

**局限**：T型团队需要时间培养——不是把不同专业的人放在一起就能自然协作。需要明确的接口规范和共同语言（如原型、交互文档）。

### 模型5：创作共同体模型
**一句话**：树洞鱼不是传统意义上的「导师管学生」的实验室，而是一个「师生共创」的创作共同体。导师的角色不是分配任务，而是提供方向、资源和反馈——学生是创作者，不是执行者。

**证据**：
- 「树洞鱼」这个昵称本身就传达了亲切、非正式的氛围 [S07]。
- DAE Lab定位「并非传统科研实验室」而是「师生的学习与创作平台」[S07]。
- 课程强调「一人一题」独立创作，学生产出自己的游戏原型 [S07]。
- 学生去向（腾讯、网易、米哈游等头部公司）说明实验室培养的是独立创作者 [S07]。

**应用场景**：用户问「实验室文化怎么建设」「怎么激发学生创作动力」「导师和学生应该是什么关系」时。

**局限**：创作共同体模式对导师的投入度要求极高，且依赖导师个人魅力和判断力。规模扩大后难以维持紧密的师生关系。

## 决策启发式（7条）

> 每条结构：规则名称 → 规则描述 → 应用场景 → 来源

### 1. 先做灰盒，再做皮囊
- **规则**：游戏原型的开发顺序——先用最简单的几何体（灰盒/Greybox）实现核心玩法并验证其可玩性，确认核心循环成立后，再逐步替换美术资源和视觉表现。反过来做（先做漂亮场景再想玩法）是最常见的失败模式。
- **来源**：私域课程中猜数字案例的教学顺序——先场景布置（功能性的）→GUI→逻辑→状态机，美术放在能跑之后 [P01]。
- **应用场景**：启动任何游戏原型时。

### 2. 两周法则
- **规则**：一个游戏原型的核心玩法验证应该在两周内完成。超过两周还没有可玩版本，大概率是范围太大了。砍掉一切不影响核心玩法的功能——菜单、存档、音效、特效、教程——先让核心循环跑起来。
- **来源**：元宇宙毕业展馆2个月完整交付的倒推——核心原型阶段应该更短 [S11][S12][S13]；猜数字案例的组织节奏 [P01]。
- **应用场景**：原型开发过程中发现进度滞后时。

### 3. 可玩比完整重要
- **规则**：在原型阶段，「3分钟的可玩体验」比「30分钟但到处是Bug的体验」有价值得多。宁可做一个只有一关但手感扎实的Demo，也不做一个有十关但每关都粗糙的半成品。
- **来源**：树洞鱼产出以「游戏原型/Demo」为定位 [S07]；12款和10款原型的快速多方向探索策略 [S14][S15]。
- **应用场景**：原型范围控制、决定砍掉什么功能时。

### 4. 每周必Build
- **规则**：原型开发期间，每周至少做一次完整的Build（打包发布），确保项目在任何时候都是「可发布」状态。最后一周才第一次Build几乎必然遇到意想不到的打包问题，导致延期。
- **来源**：全流程教学中「发布」被明确列为最后一个环节 [P01]；元宇宙展馆的WebGL发布经验 [S11][S12][S13]。
- **应用场景**：原型开发中后期。

### 5. 先定技术约束再想创意
- **规则**：给学生的项目必须先明确技术约束——用什么引擎、目标平台是什么、包体限制多大、性能预算多少。在有约束的框架内发挥创意，比「什么都可以做」更容易产出好作品。
- **来源**：元宇宙展馆首包40MB的技术约束 [S11][S12][S13]；课程中明确的技术栈（Unity+WebGL/URP）[P01]。
- **应用场景**：项目启动阶段定义范围时。

### 6. 跨学科协作的关键是「共同语言」
- **规则**：设计师和程序员之间最大的障碍不是技能差异，而是「共同语言」的缺失。解决方法是建立一个所有人都能理解的中间产物——交互流程图、纸面原型、可点击的线框图——作为沟通的基础。
- **来源**：柳喆俊跨学科背景（广告学→管理→建筑学）的亲身经验 [S01][S02]；设计工作坊的跨专业协作实践 [S07]。
- **应用场景**：跨学科团队启动协作时。

### 7. Demo做完不等于项目结束
- **规则**：原型开发完成后必须走三个步骤才算真正闭环：(1) 让至少3个「非开发者」试玩并给出反馈；(2) 整理开发过程中的技术笔记和设计决策记录；(3) 做一次公开成果展示（课堂展示/线上发布/作品集收录）。只做不展示的原型等于没做。
- **来源**：课程成果均有公开展示（元宇宙展馆全网开放、古诗文/华大成果在学院渠道报道）[S14][S15][S11][S12][S13]。
- **应用场景**：原型开发完成后。

## 原型开发实操步骤

### 从创意到可玩Demo的标准流程

```
Phase 1: 概念（1-2天）
├── 一句话定义核心体验：「玩家在做什么？为什么有趣？」
├── 纸上原型/流程图：用纸笔画出核心循环
└── 技术约束确认：引擎/平台/包体/时间

Phase 2: 灰盒原型（3-5天）
├── 用几何体搭建核心场景
├── 实现最小可玩循环（核心机制+基本输入+胜负条件）
├── 不涉及美术、音效、UI美化
└── 里程碑：自己能玩30秒不崩溃

Phase 3: 功能完善（5-7天）
├── 补充次级系统（菜单、状态机、计分）
├── 加入临时美术资源（占位素材）
├── 第一次完整Build测试
└── 里程碑：别人能玩1分钟不困惑

Phase 4: 打磨发布（3-5天）
├── 替换关键美术资源
├── 加入基础音效反馈
├── 最终Build+多平台测试
├── 3人以上外部试玩+收集反馈
└── 里程碑：链接发出去，别人能玩完
```

### 工具链偏好

| 环节 | 推荐工具 | 为什么 |
|---|---|---|
| 引擎 | Unity | 生态完整、学习资源丰富、WebGL发布成熟 |
| 版本管理 | Plastic SCM / Git | Unity官方集成好 |
| 原型工具 | 纸笔 / Figma / Miro | 在写代码前先验证想法 |
| 素材 | Unity Asset Store / 自建 | 原型阶段用免费素材，打磨阶段替换 |
| 发布 | WebGL（优先）/ PC Standalone | WebGL零安装门槛，适合快速分享 |
