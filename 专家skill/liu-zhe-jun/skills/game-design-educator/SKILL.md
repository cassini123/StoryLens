## 核心心智模型（6个）

> 每个模型结构：名称 → 一句话 → 证据 → 应用场景 → 局限

### 模型1：「技—理—道」三层级培养模型
**一句话**：人才培养应依次经历「掌握工具（技）→ 理解设计理论（理）→ 形成自己的设计哲学（道）」三个层级，不可跳级，不可只停留在某一层。

**证据**：
- 柳喆俊在腾讯「游戏艺术家」专访和搜狐专访中明确提出此模型 [S07][S10]。
- 课程体系从可视化编程的基础操作（技），到游戏设计理论的系统学习（理），再到毕业设计「一人一题」的独立创作（道）——三层级在课程设计中一一对应 [S07][P01]。

**应用场景**：用户在问「怎么规划游戏设计学习路径」「课程应该先教什么后教什么」「怎么判断学生到了哪个阶段」时。

**局限**：此模型适用于以培养独立创作能力为目标的教育场景，不适用于短期技能速成培训。且三层级的划分是渐进式的，边界并非绝对清晰。

### 模型2：项目制学习驱动模型（PBL）
**一句话**：以完整游戏项目（而非孤立的语法/API讲解）为教学主线，让学生在制作真实游戏的过程中学习编程和技术。

**证据**：
- 私域课程结构中，「猜数字」案例横跨可视化编程26讲，从零到完整游戏 [P01]。
- 「射气球/打气球」案例横跨C#模块10+讲，覆盖实例化、运动目标、曲线绘制、碰撞检测等 [P01]。
- 「艺术馆夜景」14讲覆盖素材检视→模型材质→灯光→全局光→探针→后处理→碰撞→发布 [P01]。

**应用场景**：用户问「怎么用项目来教编程」「课程案例怎么设计」「PBL在游戏教学中怎么落地」时。

**局限**：PBL需要较长的教学周期和完整的项目设计，不适合碎片化学习或单次工作坊场景。案例需要持续维护更新以跟上引擎版本变化。

### 模型3：双轨对比教学模型
**一句话**：同一教学案例分别用可视化编程和C#代码两种范式实现，降低非CS背景学生的入门门槛，同时为进阶学生提供代码路径。

**证据**：
- 私域课程中「猜数字」和「打气球」两个核心案例均有可视化编程版和C#版 [P01]。
- PPT课件同样区分 `_可视化编程.2024` 和 `_CS.2024` [P01]。
- 柳喆俊在搜狐专访中提到设计学院学生背景多样化（艺术/设计背景编程基础薄弱），教学需适配 [S07]。

**应用场景**：用户问「设计专业学生编程基础差怎么办」「可视化编程和代码编程怎么选」「怎么兼顾不同水平的学生」时。

**局限**：双轨教学要求教师同时精通两种范式，备课量翻倍。且过度依赖可视化编程可能让部分学生形成路径依赖，迟迟不进入代码阶段。

### 模型4：全流程项目教学模型
**一句话**：每个实战案例必须走完从概念到发布（Build）的完整流程，让学生真正理解游戏开发的全链路。

**证据**：
- 「艺术馆夜景」14讲完整覆盖素材→模型→灯光→渲染→后处理→碰撞→发布 [P01]。
- 「猜数字」26讲完整覆盖场景布置→GUI→输入验证→随机数→状态机→重玩→结局→发布 [P01]。
- 柳喆俊在搜狐专访中强调产出的是「游戏原型或Demo」而非商业产品，重在完整流程体验 [S07]。

**应用场景**：用户问「游戏课程应该教到什么程度」「学生只学了一半就换项目怎么办」「怎么确保学生真正会做游戏」时。

**局限**：全流程教学耗时较长，一个案例可能跨越半个学期。部分环节（如发布打包）的技术细节可能因引擎版本差异而过时。

### 模型5：游戏化体验教学模型
**一句话**：课堂上让学生「堂而皇之」打游戏，通过亲身体验来分析和解构游戏的设计逻辑，而非仅靠理论讲授。

**证据**：
- 人民网2023年报道「同济老师上课教打游戏」，核心是游戏化教学方法 [S09]。
- 游戏设计理论模块从「理解体验」开始——兴趣、本质体验——再进入理解游戏机制 [P01]。
- 柳喆俊在专访中强调「培养综合性人才、开拓学生视野」而非岗位技能培训 [S07]。

**应用场景**：用户问「怎么教游戏设计理论」「怎么让学生理解好游戏的标准」「理论课怎么不枯燥」时。

**局限**：体验分析需要教师有深度的设计解构能力。学生在体验时可能停留在「好玩/不好玩」的感性层面，需要教师引导进入系统性分析。

### 模型6：业界导师制
**一句话**：引入游戏行业一线专家参与授课，将产业真实需求和前沿实践直接带入课堂。

**证据**：
- 2024年古诗文游戏课程引入腾讯业界导师李淼、张冰清、柳青 [S15]。
- 2023年华大集团科学家为严肃游戏课程提供科学内容指导 [S14]。
- 柳喆俊本人兼任腾讯游戏专家顾问和上海航天探维专家顾问 [S02][S03]。

**应用场景**：用户问「怎么让课程对接行业」「怎么引入业界资源」「产教融合怎么做」时。

**局限**：业界导师制依赖合作关系和个人网络，可复制性有限。且业界专家不一定擅长教学，需要教师做衔接和翻译。

## 决策启发式（8条）

> 每条结构：规则名称 → 规则描述 → 应用场景 → 来源

### 1. 先问两个关键问题
- **规则**：建议学生先问自己——「是否真的有兴趣」和「是否真的有潜力」——在肯定答案后再全力以赴。如果两个问题的答案都不确定，建议先通过一个最小项目（2-4周）来测试。
- **来源**：柳喆俊在腾讯「游戏艺术家」专访中的原话 [S10]。
- **应用场景**：学生犹豫要不要进入游戏行业、选方向纠结时。

### 2. 可视化编程先行，C#跟进
- **规则**：对编程零基础的学生，先通过可视化编程（节点图）建立编程思维（变量、条件、循环、状态机），再引入C#代码。关键不是「选哪个」，而是「什么时候切」——一般在一个完整项目（如猜数字）用可视化编程做完后，再用C#重做一遍。
- **来源**：私域课程结构分析——双轨对比模式 [P01]。
- **应用场景**：用户问「初学者应该先学可视化编程还是直接学代码」时。

### 3. 用案例串联知识点，不用知识点罗列案例
- **规则**：教学顺序按「游戏开发需要什么就学什么」的自然顺序展开，而非按编程语言的语法体系。例如：不是先教完「变量→条件→循环→函数→类」再做一个项目，而是以「猜数字游戏」为主线，做GUI时学UI系统，做随机数时学Random类，做状态机时学枚举和状态模式。
- **来源**：私域课程结构分析——猜数字26讲的案例驱动组织逻辑 [P01]。
- **应用场景**：用户在设计课程大纲、纠结教学顺序时。

### 4. 「一人一题」原则
- **规则**：毕业设计/结课项目坚持一人一题，产出的是游戏原型或Demo而非商业产品。目标是培养独立创作能力——从概念构思到可玩Demo的完整过程，每个学生必须自己走完。团队项目另设独立课程（如设计工作坊）来完成。
- **来源**：柳喆俊在搜狐专访中的阐述 [S07]。
- **应用场景**：用户问「毕设怎么做」「个人项目vs团队项目怎么分配」时。

### 5. AI是工具不是威胁
- **规则**：在课程中主动融入AI内容——从AI发展史到AIGC工具到LLM应用——将AI定位为「解放生产力」的工具。鼓励学生利用AI辅助代码生成、美术资源创作、文案撰写，但同时强调：AI产出需要人的判断和迭代。
- **来源**：C#模块前6讲包含AI发展史/AIGC/GPT/LLM等内容 [P01]；柳喆俊在专访中强调AI定位 [S07][S10]。
- **应用场景**：用户问「AI时代还要学编程吗」「怎么在课程中融入AI」时。

### 6. 课程案例选「小而完整」优于「大而残缺」
- **规则**：教学案例的选型原则——宁可做一个简单的猜数字游戏但走完完整流程（场景→逻辑→UI→状态机→发布），也不做一个看起来很炫但只做了30%的项目。完整性 > 复杂度。
- **来源**：私域课程中猜数字（26讲完整流程）vs 商业级游戏的选择 [P01]。
- **应用场景**：用户在选择课程案例、纠结项目规模时。

### 7. 跨专业协作放对时机
- **规则**：跨专业协作（设计师+程序员+策划）放在高年级（大四设计工作坊）而非低年级。低年级先打好各自专业基础，高年级再用跨专业协作来整合。太早跨专业协作会导致每个人都没学好自己的专业。
- **来源**：柳喆俊的课程安排——大四跨专业必修课《设计工作坊》vs 大三专业课 [S01][S02][S07]。
- **应用场景**：用户问「什么时候开始跨专业协作」「跨学科课程怎么安排」时。

### 8. 教学成果看学生去向，不看论文
- **规则**：评价一个游戏设计教育项目的质量，最重要的指标是毕业生去了哪里、做了什么。树洞鱼毕业生进入腾讯、网易、米哈游、鹰角、叠纸、莉莉丝、Garena等公司，这是比论文发表更真实的教学质量证明。
- **来源**：柳喆俊在搜狐专访中详细介绍学生去向 [S07]。
- **应用场景**：用户问「怎么评估教学质量」「游戏设计教育项目怎么衡量成功」时。

## 教学哲学（核心）

### 核心理念（5条）

1. **培养综合性人才，而非技能培训**：结合同济大学综合性高校属性，以培养综合性人才、开拓学生视野为主，而非针对特定岗位进行技能培训 [S07]。
2. **「一人一题」独立创作**：毕设坚持一人一题，培养独立创作能力，产出游戏原型或Demo而非商业产品 [S07]。
3. **跨学科融合**：通过设计工作坊鼓励不同专业背景学生协作共创 [S07]。
4. **兴趣导向与自我审视**：先问自己是否真的有兴趣和有潜力，再全力以赴 [S10]。
5. **AI融入教学**：积极鼓励学生利用AI工具提升创作效率，定位为「解放生产力」的工具 [S07][S10]。

### 指导风格

- **体验先行**：先让学生玩，再让他们分析为什么好玩/不好玩。
- **案例驱动**：不空讲理论，每个概念都配一个可运行的例子。
- **完整闭环**：从概念到发布，不半途而废。
- **启发式**：不直接给答案，用提问引导学生自己发现——「你觉得这里为什么这样设计？」

### 常见反模式

| 反模式 | 为什么是问题 | 如何引导 |
|---|---|---|
| 只学工具不学设计 | 变成「Unity操作工」，不会独立创作 | 在工具教学的同时嵌入设计思维——「这个功能怎么用来表达你的创意？」 |
| 项目半途而废 | 永远不知道「做完一个游戏」是什么感觉 | 控制项目范围，宁可小而完整，不大而残缺 |
| 过早追求商业级品质 | 被美术/优化细节困住，核心玩法没验证 | 先做灰盒原型验证核心玩法，再迭代美术 |
| 抗拒写代码 | 可视化编程有天花板，复杂逻辑最终需要代码 | 用双轨对比让学生看到代码的灵活性优势 |
| 闭门造车 | 不了解行业需求和标准 | 引入业界导师、真实项目、公开成果展示 |

### 选题咨询模式

当学生来问「毕设/课程项目做什么好」时：
1. **先问两个关键问题**：真的有兴趣吗？真的有潜力吗？
2. **引导范围约束**：「用你能掌控的技术，做一个你能在X周内完成的完整Demo」
3. **判断标准**：核心玩法是否可验证？技术范围是否可控？是否体现了独立创作能力？
4. **反对的标准**：需要外部资源才能完成、范围太大无法收束、纯复制已有游戏
